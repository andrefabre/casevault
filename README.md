# MyDigitalLegacy Vault by Exgenesis - High-Security MVP Build

A secure, RegTech-focused, non-custodial platform that registers digital asset instructions and executes post-mortem release to nominated Executors using advanced two-key security (K1/K2).

## 🚀 Quick Start

### Prerequisites

Before you begin, ensure you have the following installed:

1. **Node.js** (v18 or later) - [Download here](https://nodejs.org/)
2. **npm** (comes with Node.js)
3. **Git** (for cloning the repository)

### Installation

1. **Clone the repository:**
   ```bash
   git clone https://github.com/andrefabre/digital-legacy-vault-MVP1.git
   cd digital-legacy-vault-MVP1
   ```

2. **Install dependencies:**
   ```bash
   npm install
   ```

3. **Set up environment variables:**
   ```bash
   cp .env.example .env
   ```
   Edit the `.env` file with your configuration.

4. **Initialize the database:**
   ```bash
   npx prisma generate
   npx prisma db push
   ```

5. **Run the development server:**
   ```bash
   npm run dev
   ```

6. **Open your browser:**
   Navigate to [http://localhost:3000](http://localhost:3000)

## 🛡️ Security Architecture

### Two-Key Security System (K1/K2)

The application implements a sophisticated two-key encryption system:

- **K1 (Owner Key)**: Generated and held by the asset owner
- **K2 (Executor Key)**: Generated by Admin and released upon death verification
- **Dual Requirement**: Both keys are required to decrypt digital asset instructions

### Security Flow

1. **Owner Phase**: Instructions encrypted with K1
2. **Storage Phase**: Encrypted data stored securely
3. **Verification Phase**: Death verification by Admin
4. **Release Phase**: K2 released to verified Executor
5. **Access Phase**: Instructions decrypted with K1 + K2

## 👥 User Roles & Access

### Owner (Default Role)
- **Access Requirements**: `is_paid = true` AND `is_kyc_verified = true`
- **Capabilities**:
  - Add encrypted digital assets
  - Nominate executor
  - Manage vault settings

### Executor
- **Access Requirements**: Nominated by Owner
- **Capabilities**:
  - Upload verification documents
  - Access released assets (when K2 available)
  - View decrypted instructions

### Admin
- **Access Requirements**: Manual role assignment
- **Capabilities**:
  - Review verification documents
  - Approve death certificates
  - Release K2 to verified executors

## 🔄 End-to-End Workflow

### 1. Owner Onboarding

### 2. Asset Management
```
Add Asset → K1 Encryption → Secure Storage → Executor Nomination
```

### 3. Executor Verification

### 4. Asset Release
```
Death Verification → K2 Assignment → K1+K2 Decryption → Instruction Access
```

## 🗄️ Database Schema

### Core Tables

1. **Users**: Authentication, roles, and security flags
2. **DigitalAssets**: Encrypted asset instructions
3. **ExecutorNominations**: Owner-to-Executor relationships
4. **VerificationDocs**: Death certificates and ID documents

### Key Fields

- `User.is_paid`: Subscription gate
- `User.is_kyc_verified`: KYC verification gate
- `User.is_mfa_setup`: MFA security gate
- `User.keyComponentK2`: Executor's secret key
- `User.release_key_K2_available`: Global K2 release flag

## 🔐 Testing the Two-Key System

### Manual Testing Flow

1. **Create Owner Account**:
   - Sign up → Setup MFA → Admin approves payment/KYC

2. **Add Digital Asset**:
   - Enter asset name and instructions
   - System encrypts with K1
   - K1 stored in browser localStorage

3. **Nominate Executor**:
   - Enter executor email
   - Status: "Pending"

4. **Executor Signup**:
   - Executor creates account with nominated email
   - Role automatically set to "Executor"

5. **Upload Verification**:
   - Executor uploads ID and death certificate
   - Documents status: "Pending"

6. **Admin Approval**:
   - Admin reviews documents
   - Admin generates and assigns K2
   - Sets `release_key_K2_available = true`

7. **Asset Access**:
   - Executor accesses vault
   - System decrypts using K1 + K2
   - Instructions displayed in clear text
